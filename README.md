# Fonbet Basketball Parser

Парсер для сбора срезов по live-баскетбольным матчам с Fonbet. Собирает игровые данные и букмекерские коэффициенты по четвертям, сохраняет в CSV для последующего обучения нейросети.

---

## Структура проекта

```
fonbet-parser/
├── pom.xml
└── src/main/java/com/fonbet/
    ├── Main.java                        ← точка входа, polling-цикл
    ├── FonbetClient.java                ← HTTP-запросы к API
    ├── FonbetParser.java                ← десериализация JSON
    ├── FonbetState.java                 ← накопление delta-состояния
    ├── SnapshotCollector.java           ← бизнес-логика сбора срезов
    ├── CsvWriter.java                   ← запись в файл
    └── model/
        ├── FonbetResponse.java          ← модель ответа API
        └── MatchSnapshot.java           ← модель одного среза
```

## Сборка и запуск

**Требования:** JDK 17+, Maven (встроенный в IntelliJ IDEA)

**Сборка:** Maven → Lifecycle → `package` (двойной клик в IntelliJ)

**Запуск:**
```bash
java -jar target/fonbet-parser-1.0-SNAPSHOT-jar-with-dependencies.jar
```

Выходной файл `basketball_snapshots.csv` создаётся рядом с jar в текущей директории.

## Как работает API Fonbet

### Endpoint
```
GET https://line-lb61-w.bk6bba-resources.com/ma/events/list
    ?lang=ru
    &version=XXXXXXXXXX
    &scopeMarket=1600
```

> **Важно:** домен `lb61` может меняться. Если парсер начнёт получать ошибки 4xx/5xx подряд — откройте DevTools на fonbet.ru → Network → Fetch/XHR и найдите актуальный хост запросов `list?lang=ru&...`. Обновите константу `BASE_URL` в `FonbetClient.java`.

### Механика delta-обновлений

API работает по принципу **delta-пакетов**:
- Первый запрос: `version=0` → получаем полный снапшот всех событий
- Каждый следующий: `version=packetVersion` из предыдущего ответа → только изменения

Это означает что пропущенный пакет (при ошибке сети) приводит к потере изменений. При 5 ошибках подряд парсер сбрасывает `version=0` и получает полное состояние заново.

### Структура ответа

```json
{
  "packetVersion": 73228158031,   // передаём как version в следующий запрос
  "sports": [...],                // сегменты/лиги (нужны для фильтрации)
  "events": [...],                // матчи и sub-events (четверти)
  "liveEventInfos": [...],        // таймер, счёт, subscores по четвертям
  "eventMiscs": [...],            // агрегированный счёт + comment с историей
  "customFactors": [...]          // коэффициенты по каждому событию
}
```

### Как устроены события

Каждый матч — это дерево событий:
- **Root-событие** (`level=1`, `kind=1`) — сам матч с командами
- **Sub-events** (`level=2`, `kind=100401/100402/100403/100404`) — четверти

```
Матч (id=63549287, level=1)
├── 1-я четверть (id=63549288, level=2, kind=100401, parentId=63549287)
├── 2-я четверть (id=63549289, level=2, kind=100402, parentId=63549287)
└── ...
```

Коэффициенты тотала по четверти находятся в `customFactors` sub-event'а четверти, а не самого матча.

### Как найти тотал четверти

В `customFactors[e=<sub_event_id>].factors`:
- `f=930` — тотал Больше
- `f=931` — тотал Меньше
- `pt` — линия тотала в виде строки, например `"28.5"`
- `v` — коэффициент

**Признак четвертного тотала:** `pt < 100`. Матчевые тоталы имеют `pt > 150`.

### Как определить текущую четверть

Из `liveEventInfos[eventId=<match_id>]`:
1. **Через `subscores`** — содержит только активные периоды. `kindId=100401` → Q1, `100402` → Q2 и т.д.
2. **Через `scores[1]`** — массив счёта по четвертям с `title="четверть"`. Количество элементов = номер текущей четверти.

### Как определить итог завершённой четверти

Три стратегии (в порядке приоритета):

1. **`liveEventInfo.scores[1][quarterIndex]`** — после завершения Q1 её финальный счёт фиксируется в этом массиве
2. **Кеш `lastKnownQuarterScore`** — парсер сохраняет каждый известный счёт четверти; используется если API уже обнулил `subscores` к моменту детекции смены четверти
3. **`eventMisc.comment`** — строка вида `"(27-22 14-18)"` содержит счёт завершённых четвертей через пробел

## Логика сбора срезов

### Polling
Каждые **5 секунд** запрашиваем delta-обновление. Интервал задаётся константой `POLL_INTERVAL_MS` в `Main.java`.

### Снятие среза
Для каждого активного матча в Q1 срез снимается не чаще одного раза в **30 секунд** (`SNAPSHOT_INTERVAL_MS` в `SnapshotCollector.java`). Если за 30 секунд коэффициенты изменились несколько раз — все изменения будут в следующем срезе.

### Двухэтапная запись
1. Срез снимается → хранится в памяти (`pending`)
2. После завершения четверти → в каждый pending-срез дописывается `quarter_result` → все срезы записываются в CSV

При остановке (`Ctrl+C`) — незавершённые срезы записываются с `quarter_result=N/A`.

### Смена четверти
Детектируется сравнением `activeQuarter > prevQuarter` для каждого матча. Если матч пропал из live (завершился или снят) — pending-срезы записываются с `N/A`.

## Фильтрация матчей

Парсер пропускает:

| Что отфильтровывается | Как определяется |
|---|---|
| Киберспорт (NBA 2K, H2H, EsportsBattle) | Название сегмента содержит: `2k`, `esport`, `h2h`, `virtual`, `gg league` |
| Киберспорт (запасной фильтр) | Имя команды содержит ник игрока в скобках: `Торонто (CARNAGE)` |
| Женский баскетбол | Название сегмента содержит: `женщин`, `women`, `ladies` |
| Женский баскетбол (запасной фильтр) | Имя команды содержит `(ж)` |
| Незапущенные матчи | `place != "live"` или таймер пустой |
| Sub-events | `level != 1` |

## Формат CSV

```
timestamp,match_id,team1,team2,total_score1,total_score2,
quarter_score1,quarter_score2,quarter_number,minute,
over_odds,under_odds,line,quarter_result
```

**Пример строки:**
```
2026-03-26 11:47:05,63549287,Рейвенс,Октопус,3,2,3,2,1,1:08,1.80,1.90,61.5,53 (30-23)
```

| Поле | Описание |
|---|---|
| `timestamp` | Время снятия среза |
| `match_id` | ID матча в системе Fonbet |
| `team1`, `team2` | Названия команд |
| `total_score1/2` | Общий счёт матча на момент среза |
| `quarter_score1/2` | Счёт текущей четверти на момент среза |
| `quarter_number` | Номер четверти (1-4) |
| `minute` | Таймер на момент среза, например `3:45` |
| `over_odds` | Коэффициент тотал Больше по четверти |
| `under_odds` | Коэффициент тотал Меньше по четверти |
| `line` | Линия тотала, например `28.5` |
| `quarter_result` | Итог четверти: `53 (30-23)` или `N/A` |

> `quarter_result` формат: `<сумма> (<команда1>-<команда2>)`, например `53 (30-23)` означает 53 очка в четверти, 30 у первой команды и 23 у второй.

## Настройки

Все ключевые константы:

| Файл | Константа | По умолчанию | Описание |
|---|---|---|---|
| `Main.java` | `POLL_INTERVAL_MS` | 5000 | Интервал polling'а (мс) |

| `FonbetClient.java` | `BASE_URL` | `https://line-lb61-w...` | Endpoint API |

### Сбор всех четырёх четвертей

В `SnapshotCollector.java` замените:
```java
private static final Set<Integer> TARGET_QUARTERS = new HashSet<>(
        Collections.singletonList(1)
);
```
на:
```java
private static final Set<Integer> TARGET_QUARTERS = new HashSet<>(
        Arrays.asList(1, 2, 3, 4)
);
```

## Известные особенности

**`Q1 ?` и `score 0-0` в начале матча** — первые несколько секунд после появления матча в live таймер ещё не приходит в API. Парсер пропускает такие срезы (`timer == null`).

**`result=N/A`** — возникает если:
- Матч пропал из live до завершения четверти (снят, технические проблемы)
- API не успел передать финальный счёт ни через один из трёх каналов

**Смена домена** — Fonbet периодически меняет номер балансировщика (`lb61` → `lb62` и т.д.). Симптом: ошибки HTTP 4xx. Решение: обновить `BASE_URL` в `FonbetClient.java`.

**Рост памяти** — поле `events` в `FonbetState` накапливает все события (завершённые матчи не удаляются). За несколько часов работы накапливается ~10-11k событий, что нормально и не влияет на производительность.
